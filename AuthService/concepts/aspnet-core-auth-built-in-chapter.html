Last chapter: the built-in authentication stack

Video: ASP.NET Core Authentication (.NET 7 Minimal APIs C#) — Raw Coding · Chapter starts ~16:47 (same, t=1007s)

Earlier in the video you manually issued a cookie, used Data Protection, refactored into an auth service, and ran custom middleware that populated HttpContext.User. This final segment replaces that hand-rolled code with the framework primitives and traces how they connect under the hood.

What you register in Program.cs

Services

builder.Services.AddAuthentication() · .AddCookie(...)

  • Selects an authentication scheme (here: cookie). Think “who issued my identity” — government ID vs driving licence vs job centre, but for HTTP auth.
  • Registers handlers and supporting services (cookie format, chunking, validation, ticket serialization).
Middleware

app.UseAuthentication()

  • Equivalent to the custom middleware that read the cookie and set HttpContext.User.
  • Runs before your endpoints; resolves the user for the rest of the pipeline.
Signing in — Replace manual cookie writes with await HttpContext.SignInAsync(CookieAuthenticationDefaults.AuthenticationScheme, claimsPrincipal). Build a ClaimsPrincipal / ClaimsIdentity (with the same scheme) representing the user you want to persist in the cookie. Your database lookup and password check still happen before this call; you map the stored user to claims, then sign in.

Request pipeline (conceptual)

Authentication middleware runs before your route handlers so endpoints see a populated user when a valid cookie is present.

Services and handlers (the “spine”)

The presenter opens SignInAsync in the framework and follows it to IAuthenticationService. That service depends on an authentication handler provider, which resolves the correct handler for the scheme (cookie, JWT bearer, Google, etc.).

Your earlier custom code split into “dealing out” and “recognizing” a session; the handler is where those behaviors live for the cookie scheme.

Two flows: sign-in vs authenticate

Sign-in (login success)

  1. Validate credentials (your code).
  2. Build ClaimsPrincipal.
  3. SignInAsync → authentication service → cookie handler’s HandleSignInAsync.
  4. Encrypted ticket written to the browser; no server session unless you plug in ITicketStore.

Authenticate (each request)

  1. UseAuthentication() runs.
  2. Default scheme determines which handler runs.
  3. Handler’s HandleAuthenticateAsync reads the cookie, decrypts/validates, builds ClaimsPrincipal.
  4. Principal attached to HttpContext.User for downstream code.

Same handler type, different entry points: one writes the cookie after login; the other reads it on subsequent requests.

Important mental models from this chapter

  • No mystery server memory — By default the signed-in state lives in the browser cookie as a protected ticket. The server proves authenticity by decrypting/validating what it previously issued (Data Protection–style pipeline inside the cookie stack).
  • Optional server-side tickets — If you need sessions in a database or distributed cache, configure an ITicketStore when registering cookie authentication; otherwise there is no backing store per login.
  • Schemes compose — You can register cookie + JWT + external providers; the default scheme tells middleware which handler to use when one is not specified explicitly.
  • Testing and mocking — Knowing that IAuthenticationService delegates to scheme-specific handlers helps you decide what to substitute or spy on in tests.
Scope reminder from the video — Cookie auth still has many edge cases (SameSite, persistence, size limits, security headers, etc.). This chapter maps where framework code runs, not every production hardening step.
